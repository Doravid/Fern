## What did I learn?

### Buffer Overflows

Lecture challenges are, as the name implies, challenges we go over during the lecture. These started out very simple. For example, you have a call to the function gets on a buffer right next to a variable x. You win the challenge if `x != 0`. Then the professor explains how gets works, how it doesn't check the size of the array its reading into and how it will write variables after the buffer on the stack. Then we have to solve the problem on own time (if we didn't follow along in class which I never did lol). 

We then moved into setting individual bytes, like what if instead of x != 0 it was x == 512. This required us to learn about the byte layout of ints, but otherwise was not really that much harder. 

### Control Flow Hijacking

Then we learned about control flow hijacking! Basically if you know the address of a function you want to jump to, lets say a function called hax_your_comput0r(), and you are able to write to the stack, again with gets, then you can overwrite the return address of the function, with the address of your hax_your_comput0r function. This lets you jump to any function you know the address of. However this technique depends on you being able to change the return address.

The next thing we learned about was Stack Canaries. They are just numbers put between the stack and the return address, and if it is modified, then the program will instantly crash. This makes it a lot harder to return where you want to. To get around this you either need to guess the value, read the value, or write directly to the return address, e.g without modifying the canary. 

All of these are very difficult in practice but depending on the challenge it might be possible. For example, one challenge we had to do involved a program what constantly forked itself, then called gets. Fun fact: When a program forks itself, its stack canary is the same as the program it was forked from. You could guess the canary one byte at a time and then use it on the main process to jump to your win function!

### Ret2LibC

Another type of exploit we learned about was [ret2libc](https://medium.com/@mounisha.makineni12/return-to-libc-attack-exploiting-buffer-overflow-for-privilege-escalation-105fea0fe9a2) which is essentially just jumping to a win function inside of the C standard library called `system`. System will spawn a shell and run the command passed to it. It does require setting up an argument with the text `/bin/sh` so it's not a true win function, but it is still very powerful. 

We had a whole short lecture series on printf(). I am now terrified of printf... never pass a buffer straight into printf. e.g print(my_array). This is basically instantly making your program compromised as it lets you read/write stack values due to printf having wayyy too many features.
